# 需求 — v1.1 AA Foundation

**Goal:** 让 ceres_wallet_core 给上层 dapp / 钱包应用提供完整的 ERC-4337 / EIP-7702 账户抽象 mid-layer 原语。secp256k1 与 P-256 passkey 两类 owner 通过统一签名接口暴露；Barz SmartAccount 的反事实地址、initCode、UserOperation 组装、ERC-1271 消息签名、链上签名格式化、EIP-7702 EOA 升级一站式覆盖。

**Boundary:** 纯 Dart 组合层。Bundler RPC、paymaster RPC、平台 passkey 调用都不在 core 库。

---

## 签名抽象（Signer Abstraction）
- [x] **AA-01**: 提供 `EvmSigner` 抽象接口与 sealed `EvmSignature` 联合类型。`Future<EvmSignature> signDigest(Uint8List digest32)` 唯一签名入口，单飞（single-flight）保证。 ✓ v1.1
- [x] **AA-02**: 提供 `Secp256k1Signer`（包 `TWPrivateKey`）— 65 字节 r‖s‖v 输出，可与 `EvmSigner` 接口互操作。 ✓ v1.1
- [x] **AA-03**: 提供 `PasskeySigner` — 接收上层注入的 `Future<PasskeyAssertion> Function(Uint8List)` 适配器，组合 `TWBarz.getFormattedSignature` 输出 ABI 编码的 Barz signature blob（约 290 字节）。 ✓ v1.1

## ERC-4337 UserOperation 组装
- [x] **AA-04**: 提供 `Erc4337Builder` — 单一 Dart 类，`v06(...)` / `v07(...)` 命名构造器分别对应 `Ethereum.UserOperation` 与 `Ethereum.UserOperationV0_7` 两套 proto；跨版本字段混用必须 `throw StateError`。 ✓ v1.1
- [x] **AA-05**: `Erc4337Builder.attachSignature(EvmSignature)` 是签名→bytes 转换的唯一入口；passkey path 强制走 `getFormattedSignature` 并断言长度 ≥ 200 字节。 ✓ v1.1
- [x] **AA-06**: 提供 `Erc4337Calldata.executeCall(target, value, data)` 与 `Erc4337Calldata.executeBatch(targets, values, datas)` — ABI 编码 Barz / 标准 SmartAccount inner-call calldata；selector 常量集中管理并通过 keccak256 测试验证。 ✓ v1.1

## 反事实地址与部署
- [x] **AA-07**: 提供 `PasskeyBarzAddress.compute(deployment, p256PubKey, salt)` — Dart-friendly 反事实地址派生；内部用 `TWBarz.getCounterfactualAddress` 派生后，**强制** 用 `TWEthereumEip1014.create2Address` 做 round-trip 校验，不一致则 throw。 ✓ v1.1
- [x] **AA-08**: 提供 `BarzDeployment` 不可变 value 对象 + 内置 per-chain registry，至少覆盖 ETH mainnet / Sepolia / Base / Arbitrum / Optimism / Polygon。每条记录含 (chainId, factory, verificationFacet, entryPointAddress, entryPointVersion)。 ✓ v1.1
- [x] **AA-09**: 提供 `BarzInitCode.forPasskey(deployment, p256PubKey, salt)` — 同时生成 v0.6 monolithic `initCode` 与 v0.7 split `(factory, factoryData)`。 ✓ v1.1

## ERC-1271 消息签名
- [x] **AA-10**: 提供 `Erc1271Helper`（per-(barzAddress, chainId) 实例）— `personalSignDigest(message)` / `typedDataDigest(typedData)` / `formatPasskeySignature(assertion)` / `encodeIsValidSignature(hash, sig)` 四个方法；不缓存 domain（避免跨链/跨账户混用）。 ✓ v1.1
- [x] **AA-11**: `Erc1271Helper` 文档明确："本 helper 输出经 `IERC1271.isValidSignature` 验证的签名，**不**适用于 EOA `ecrecover` 路径"。 ✓ v1.1

## EIP-7702 EOA 升级
- [x] **AA-12**: 提供 `Eip7702Upgrader.buildAuthorization(chainId, contractAddress, nonce, signer)` — 包装 `TWBarz.signAuthorization`，输出可附在 EIP-7702 交易里的 signed authorization tuple；signer 可以是 `Secp256k1Signer` 也可以是 `PasskeySigner`（passkey path 会拒绝并 throw — 7702 必须 secp256k1）。 ✓ v1.1

## 高级（P2 — v1.1.x 增量）
- [x] **AA-13**: 提供 `BarzDiamondCut.encode(adds, removes, replaces)` — 生成 Barz diamond facet 升级 calldata（add / remove / replace facet 三种动作）。 ✓ v1.1
- [x] **AA-14**: 提供 `BarzSessionKey.installCalldata(key, validUntil, allowedTargets)` — 生成 Barz session key 安装 calldata，包装 facet ABI 调用。 ✓ v1.1
- [x] **AA-15**: 提供 `PasskeyBarzAddress.acrossChains(p256PubKey, salt, chainIds)` — 一次返回一个 passkey 在多条链上各自的反事实地址（`Map<int chainId, String address>`）。 ✓ v1.1

## Out of Scope（明确排除）

- Bundler RPC client（`eth_sendUserOperation`、`eth_estimateUserOperationGas`、`eth_getUserOperationReceipt`）— 属上层 dapp 网络层
- Paymaster RPC（`pm_sponsorUserOperation` 等）— 属上层 dapp
- 平台 passkey 调用（iOS `ASAuthorizationController` / Android Credential Manager）— 属独立 Flutter plugin
- Gas oracle / fee suggestion — 属上层
- ERC-7579 modular accounts — defer to v2+
- ERC-7677 paymaster envelope — defer to v2+
- Sepolia E2E 链上联调作为 release gate — v1.1.0 不强制（可补 v1.2）
